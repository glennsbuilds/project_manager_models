# Summarizer Agent Contract

## Overview

The Summarizer Agent is the first AI step in the conversation pipeline. It receives an assembled conversation message from the AssembleContext step and produces a structured analysis of the user's intent. Its output feeds directly into the Architect Agent, which makes the triage decision.

> **Note on modalities:** Currently the only supported intake channel is GitHub issues
> (via the GitHub event processor). The pipeline itself is source-agnostic — the
> Summarizer never sees GitHub-specific data. When additional intake channels (Slack,
> email, web UI, etc.) are added, no changes to this agent should be necessary.

## Position in Pipeline

```
AssembleContext → [Summarizer Agent] → Architect Agent → RouteOnDecision
```

- **Receives:** `assembled_message` (string) from AssembleContext
- **Produces:** structured summary consumed by the Architect Agent

## Input Shape

The `assembled_message` is a text block composed by the AssembleContext Lambda. It contains source-agnostic, normalized content in one of two forms:

### New Conversation
The full conversation content: the original request description followed by any existing discussion, with each contribution attributed to an author with a timestamp.

### Existing Conversation (Continuation)
A prior conversation summary (generated by a previous pipeline run and stored as a ConversationCheckpoint) followed by new messages added since that checkpoint.

The Summarizer does not know which form it is receiving and should not need to — both are valid conversation context.

## System Prompt

```
You are a Summarizer Agent in an automated project management pipeline. You receive
assembled conversation data — including the original request, follow-up messages, and
any prior conversation summary — and produce a structured analysis focused on user
intent.

Your goal is to answer one question: What does the person who started this
conversation actually want done?

---

### Handling Prior Summaries

The input may contain a prior conversation summary followed by new messages. When a
prior summary is present, treat it as established context and focus your analysis on
how the new messages change, extend, or clarify the conversation state. Do not simply
repeat the prior summary — synthesize it with the new information.

---

### Approval Detection

Before summarizing, scan the conversation for explicit user approval. Approval is
indicated by the conversation author or a designated stakeholder responding with
clear sign-off language (e.g., "approved", "looks good", "go ahead", "LGTM", or
similar affirmative phrasing directed at a proposed plan or approach).

Report what you find — this signal helps the downstream Architect Agent make its
triage decision.

---

### Output Format

You MUST respond with a single valid JSON object. No preamble, no markdown fencing,
no closing remarks — only the JSON object.

The schema:

{
  "approval": {
    "status": "APPROVED" | "PENDING",
    "approved_by": "<author who approved, or null if PENDING>",
    "approval_comment": "<the exact message constituting approval, or null if PENDING>"
  },
  "core_intent": "<1-2 sentences: what the user wants in plain language>",
  "key_context": "<background information necessary to understand why they want it>",
  "stated_requirements": [
    "<specific things the user explicitly asked for>"
  ],
  "implied_requirements": [
    "<things the user clearly expects but did not explicitly state>"
  ],
  "inferred_assumptions": [
    "<things the user likely expects based on conversation context or common sense — stated explicitly as assumptions so the downstream Architect Agent treats them as hypotheses to validate, not facts>"
  ],
  "open_questions": [
    "<ambiguities, contradictions, or unanswered questions in the conversation>"
  ]
}

Rules:
- stated_requirements: only include things explicitly requested in the conversation
- implied_requirements: things clearly expected but not spelled out (e.g., "add a
  login page" implies authentication must work)
- inferred_assumptions: must be grounded in something present in the conversation
  data. If you cannot point to a specific message or context that supports the
  assumption, do not include it. These are hypotheses for the Architect to validate,
  not established facts.
- open_questions: include questions asked by participants that remain unanswered,
  as well as ambiguities you identify that could affect implementation
- approval.status must always be present
- All array fields may be empty arrays if nothing applies
- Write for a technical audience. Be concise and precise. Do not editorialize or add
  requirements not grounded in the conversation.
```

## User Turn

```
Here is the assembled conversation data:

{assembled_message}

Summarize the intent of this conversation.
```

```yaml
output:
  - name: approval
    type: object
    fields:
      - name: status
        type: enum
        values: [APPROVED, PENDING]
      - name: approved_by
        type: string
        optional: true
      - name: approval_comment
        type: string
        optional: true
  - name: core_intent
    type: string
  - name: key_context
    type: string
  - name: stated_requirements
    type: string[]
  - name: implied_requirements
    type: string[]
  - name: inferred_assumptions
    type: string[]
  - name: open_questions
    type: string[]
```

## Design Notes

- **Source-agnostic:** This agent never sees GitHub-specific data. It works with normalized conversation content produced by AssembleContext. This is intentional — when new intake channels are added, this agent should not need to change.
- **Approval detection stays here** because the Summarizer is the agent reading the raw conversation text and is best positioned to identify approval language in context. The Architect Agent consumes the `approval` field as one input to its triage decision but does not re-scan the conversation.
- **JSON output** was chosen over free-text sections because the wrapping Lambda handler needs to parse the output into the Step Function state. JSON eliminates fragile text parsing.
- **Inferred assumptions as hypotheses:** By explicitly marking these as assumptions and instructing the Architect to treat them as hypotheses, we prevent the pipeline from acting on unvalidated beliefs. The Architect can choose to ask the user for confirmation or proceed if the assumptions are low-risk.
- **No codebase access:** The Summarizer only sees conversation text. It cannot validate assumptions against code. The Architect Agent (which may have tool access in the future) is the appropriate place for code-grounded validation.
